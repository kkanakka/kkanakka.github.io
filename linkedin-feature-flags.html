Core Design Principles

  • Edge-First Evaluation — Flag checks never touch the network. Services evaluate against local cache.
  • Zero Single Points of Failure — Even if control plane dies, services keep running with last known state.
  • Delta Streaming — Only changes propagate, not full snapshots. Bandwidth scales with change rate, not flag count.
  • Consistent User Hashing — A user never flips cohorts between requests within a session.
  • Global Kill Switch — Disable any flag across entire LinkedIn fleet in under 10 seconds.
  • Immutable Audit Trail — Hash-chained audit log for compliance and debugging.

Five Core Components

  1. Control Plane — Where engineers define flags and targeting rules (UI + REST API)
  2. Metadata Store — Durably holds all flag versions and targeting rules (Espresso)
  3. Distribution Plane — Fans changes out to the edge using delta streaming (Kafka)
  4. Edge Evaluators — In-memory caches co-located with every service (embedded SDK)
  5. Audit & Metrics Service — Maintains immutable hash-chained log (Kafka + Espresso)

Edge SDK Implementation

Embedded Library: Zero-network evaluation with local caching

// LinkedIn Java SDK
public class LinkedInFeatureFlags {
    private final ConcurrentHashMap<String, FlagConfig> flagCache;
    private final KafkaConsumer<String, FlagEvent> consumer;
    private final MemcacheClient memcache;
    
    // ≤2ms evaluation - pure in-memory hash lookup
    public boolean isEnabled(String flagKey, String userId) {
        FlagConfig flag = flagCache.get(flagKey);
        if (flag == null) return false;
        
        // Consistent hashing for stable cohorts
        long hash = hash(flagKey + userId);
        return (hash % 100) < flag.getRolloutPercentage();
    }
    
    // Async background thread processes Kafka deltas
    private void processKafkaDeltas() {
        while (running) {
            ConsumerRecords<String, FlagEvent> records = consumer.poll(100);
            for (ConsumerRecord<String, FlagEvent> record : records) {
                updateLocalCache(record.value());
            }
        }
    }
}

User Cohort Assignment Algorithm

Consistent Hashing Implementation

// Ensures stable user assignment across sessions
public class CohortAssignment {
    
    // SHA-256 based consistent hashing
    public static boolean isUserInCohort(String flagKey, String userId, int rolloutPercentage) {
        // Use flag key + user ID to ensure stability
        String hashInput = flagKey + ":" + userId;
        
        // SHA-256 for uniform distribution
        byte[] hash = MessageDigest.getInstance("SHA-256").digest(hashInput.getBytes());
        
        // Convert first 8 bytes to long
        long hashLong = ByteBuffer.wrap(hash).getLong();
        
        // Modulo 100 for percentage bucket
        int bucket = (int) Math.abs(hashLong % 100);
        
        return bucket < rolloutPercentage;
    }
    
    // Support for multi-variate experiments
    public static String getVariant(String flagKey, String userId, String[] variants, int[] weights) {
        String hashInput = flagKey + ":" + userId;
        byte[] hash = MessageDigest.getInstance("SHA-256").digest(hashInput.getBytes());
        long hashLong = ByteBuffer.wrap(hash).getLong();
        
        int bucket = (int) Math.abs(hashLong % 100);
        int cumulative = 0;
        
        for (int i = 0; i < variants.length; i++) {
            cumulative += weights[i];
            if (bucket < cumulative) {
                return variants[i];
            }
        }
        
        return "control"; // default fallback
    }
}

Why Both Local Cache AND Memcache Are Needed

The Bootstrap Problem

Without Memcache, new services would have a "cold start" problem:

❌ Without Memcache:
New Service Starts → Empty local cache → Wait for Kafka → Gradually receive flags
↑ Could take minutes to get all flags
✅ With Memcache:
New Service Starts → Load from Memcache → Full flag set in 100ms → Start Kafka for updates

Bootstrap & Warm Cache Strategy

New services bootstrap from Memcache before establishing Kafka connection

// Service startup sequence
public class FlagSDKInitializer {
    
    public void initialize() {
        // Step 1: Bootstrap from Memcache (warm cache)
        Map<String, FlagConfig> bootstrapFlags = loadFromMemcache();
        flagCache.putAll(bootstrapFlags);
        
        // Step 2: Connect to Kafka for real-time updates
        startKafkaConsumer();
        
        // Step 3: Background sync to catch any missed deltas
        schedulePeriodicSync();
    }
    
    private Map<String, FlagConfig> loadFromMemcache() {
        // Fetch serialized flag snapshot from Memcache
        byte[] snapshot = memcacheClient.get("feature_flags:snapshot");
        return deserializeFlagSnapshot(snapshot);
    }
    
    // Memcache gets updated every 30s by Control Plane
    private void updateMemcacheSnapshot() {
        Map<String, FlagConfig> allFlags = espressoClient.getAllFlags();
        byte[] serialized = serializeFlagSnapshot(allFlags);
        memcacheClient.set("feature_flags:snapshot", serialized, 300); // 5min TTL
    }
}
  • Cache Key Strategy — feature_flags:snapshot contains full flag state
  • TTL — 5 minutes (shorter than Kafka retention for safety)
  • Size — ~2MB for 10,000 flags (compressed with Snappy)
  • Refresh Pattern — Control Plane updates every 30 seconds

Gradual Rollout Implementation

Automated Metric Gates

Integration with LinkedIn's metrics infrastructure for automated rollout control

// Rollout controller with metric gates
public class GradualRolloutController {
    
    // Defined rollout stages with safety gates
    private final int[] ROLLOUT_STAGES = {1, 5, 25, 50, 100};
    private final String[] METRIC_GATES = {
        "error_rate_lt_0.1_percent",
        "p99_latency_lt_500ms", 
        "success_rate_gt_99.9_percent"
    };
    
    public void executeGradualRollout(String flagKey) {
        for (int i = 0; i < ROLLOUT_STAGES.length; i++) {
            int stage = ROLLOUT_STAGES[i];
            
            // Update rollout percentage
            updateFlagRollout(flagKey, stage);
            
            // Wait for propagation
            Thread.sleep(Duration.ofMinutes(5));
            
            // Check metric gates
            if (!allMetricGatesPassing(flagKey)) {
                // Auto-rollback on metric violation
                rollbackFlag(flagKey);
                alertOnCall("Rollout failed metric gates for " + flagKey);
                break;
            }
            
            // Log progress
            auditLog.info("Flag {} advanced to {}% rollout", flagKey, stage);
        }
    }
    
    private boolean allMetricGatesPassing(String flagKey) {
        for (String gate : METRIC_GATES) {
            if (!metricsClient.evaluateGate(gate, flagKey)) {
                return false;
            }
        }
        return true;
    }
}

Kill Switch Implementation

Global Emergency Disable

10-second global flag disable across all LinkedIn datacenters

// Kill switch with cross-DC propagation
public class KillSwitchService {
    
    // Emergency disable flag across all DCs
    public void emergencyDisableFlag(String flagKey, String reason, String operator) {
        KillSwitchEvent event = KillSwitchEvent.builder()
            .flagKey(flagKey)
            .action("EMERGENCY_DISABLE")
            .reason(reason)
            .operator(operator)
            .timestamp(System.currentTimeMillis())
            .build();
        
        // Broadcast to all DC Kafka topics simultaneously
        List<String> dcTopics = Arrays.asList(
            "feature-flags-ltx1",
            "feature-flags-lva1", 
            "feature-flags-ei4",
            "feature-flags-grid2"
        );
        
        // Send to all DCs in parallel for speed
        dcTopics.parallelStream().forEach(topic -> {
            kafkaProducer.send(topic, flagKey, event);
        });
        
        // Also update Espresso for durability
        espressoClient.disableFlag(flagKey, reason);
        
        // Log for audit trail
        auditLogger.emergency("KILL_SWITCH activated for {} by {} reason: {}", 
                             flagKey, operator, reason);
    }
    
    // Priority processing in Edge SDKs
    public void processKillSwitchEvent(KillSwitchEvent event) {
        if ("EMERGENCY_DISABLE".equals(event.getAction())) {
            // Immediately disable in local cache
            flagCache.put(event.getFlagKey(), 
                         FlagConfig.disabled(event.getFlagKey()));
            
            // Skip normal processing queue - handle immediately
            logger.warn("KILL SWITCH: Disabled {} - {}", 
                       event.getFlagKey(), event.getReason());
        }
    }
}

Audit Trail Implementation

Hash-Chained Compliance Log

// Immutable audit records with hash chaining
public class AuditService {
    
    // Every flag change produces signed audit record
    public void recordFlagChange(String flagKey, FlagConfig oldConfig, FlagConfig newConfig, String operator) {
        AuditRecord record = AuditRecord.builder()
            .recordId(UUID.randomUUID().toString())
            .flagKey(flagKey)
            .operation(determineOperation(oldConfig, newConfig))
            .oldValue(oldConfig != null ? serialize(oldConfig) : null)
            .newValue(serialize(newConfig))
            .operator(operator)
            .timestamp(System.currentTimeMillis())
            .previousRecordHash(getLastRecordHash(flagKey))
            .build();
        
        // Calculate hash including previous record for chaining
        String recordHash = calculateRecordHash(record);
        record.setRecordHash(recordHash);
        
        // Store in Kafka for durability and Espresso for queries
        kafkaProducer.send("audit-log", flagKey, record);
        espressoClient.storeAuditRecord(record);
    }
    
    private String calculateRecordHash(AuditRecord record) {
        String hashInput = record.getFlagKey() + ":" + 
                          record.getOperation() + ":" + 
                          record.getNewValue() + ":" + 
                          record.getTimestamp() + ":" + 
                          record.getPreviousRecordHash();
        
        return DigestUtils.sha256Hex(hashInput);
    }
    
    // Verify audit chain integrity
    public boolean verifyAuditChain(String flagKey) {
        List<AuditRecord> records = espressoClient.getAuditHistory(flagKey);
        
        for (int i = 1; i < records.size(); i++) {
            AuditRecord current = records.get(i);
            AuditRecord previous = records.get(i - 1);
            
            if (!current.getPreviousRecordHash().equals(previous.getRecordHash())) {
                return false; // Chain broken
            }
        }
        
        return true;
    }
}

Page 4 — Reliability, Failure Scenarios & Cross-Colo Design

Failure Scenarios & Mitigation Strategies

Scenario 1: Control Plane Outage

Failure: React UI and REST API become unavailable

Impact: Engineers cannot create/modify flags, but all existing flags continue evaluating

Mitigation:

  • Edge SDKs continue evaluating from local cache (no dependency on Control Plane)
  • Multi-AZ deployment with health checks and automatic failover
  • Read-only mode available from any DC if master is down
  • Emergency kill-switch available via direct Kafka publish

Recovery Time: 0 seconds (transparent to flag evaluation)

Scenario 2: Espresso Database Failure

Failure: Primary Espresso shard becomes unavailable

Impact: Cannot persist new flags, but reads continue from replica

Mitigation:

  • Automatic failover to read replica in same DC (30s RTO)
  • Cross-DC failover to LVA1 if LTX1 Espresso fully down (2 min RTO)
  • Memcache bootstrap ensures new services can still start
  • Kafka replay queue allows catch-up after recovery

Recovery Time: 30 seconds (automatic), 2 minutes (cross-DC)

Scenario 3: Kafka Partition Failure

Failure: Kafka topic becomes unavailable or messages are lost

Impact: Flag changes don't propagate to Edge SDKs

Mitigation:

  • Kafka replication factor of 3 prevents single-broker failure
  • Cross-DC Kafka mirroring for disaster recovery
  • SDKs poll Memcache every 5 minutes as backup propagation path
  • Manual "resync-all" command forces full cache refresh
  • Flag drift monitoring alerts within 2 minutes of propagation failure

Recovery Time: 0 seconds (replication), 5 minutes (Memcache fallback)

Scenario 4: Flag Drift (20% of Europe out of sync)

Failure: Network issues cause subset of Edge SDKs to miss updates

Impact: Inconsistent flag behavior across user sessions

Detection:

  • Checkly monitoring probes flag evaluation from multiple regions
  • SDKs report cache checksum every 60 seconds
  • Drift alert fires when >5% of nodes disagree on flag state

Mitigation:

  • Automatic "resync-all" triggered when drift >10%
  • Individual SDK reset via management endpoint
  • Emergency broadcast channel bypasses Kafka

Recovery Time: 30 seconds (resync-all), 10 seconds (emergency broadcast)

Scenario 5: Memcache Cluster Down

Failure: Memcache cluster becomes unavailable

Impact: New services cannot bootstrap flag cache

Mitigation:

  • Services fall back to direct Espresso query for bootstrap
  • Degraded performance during startup but no outage
  • Multi-AZ Memcache deployment with automatic failover
  • Local disk cache backup on each service host

Recovery Time: 0 seconds (automatic fallback), 60 seconds (Espresso bootstrap)

Cross-DC Failover Implementation

// Datacenter failover controller
public class CrossDCFailoverController {
    
    private enum DCStatus { HEALTHY, DEGRADED, FAILED }
    private final Map<String, DCStatus> dcHealthMap = new ConcurrentHashMap<();
    
    // Continuous health monitoring
    @Scheduled(fixedDelay = 30000) // every 30s
    public void monitorDatacenterHealth() {
        for (String dc : Arrays.asList("ltx1", "lva1", "ei4", "grid2")) {
            DCStatus status = checkDatacenterHealth(dc);
            DCStatus oldStatus = dcHealthMap.put(dc, status);
            
            if (status == DCStatus.FAILED && oldStatus != DCStatus.FAILED) {
                handleDatacenterFailure(dc);
            }
        }
    }
    
    private void handleDatacenterFailure(String failedDC) {
        switch (failedDC) {
            case "ltx1":
                // Primary DC failed - promote LVA1 to primary
                promoteSecondaryToPrimary("lva1");
                updateDNSRouting("feature-flags.linkedin.com", "lva1");
                break;
                
            case "lva1":
                // Hot standby failed - update monitoring
                alertOncall("LVA1 datacenter failed - no hot standby available");
                break;
                
            default:
                // Other DC failures are less critical
                alertOncall(failedDC + " datacenter failed - services degraded");
        }
    }
    
    private DCStatus checkDatacenterHealth(String dc) {
        try {
            // Check Kafka cluster health
            boolean kafkaHealthy = kafkaHealthCheck(dc);
            // Check Espresso cluster health  
            boolean espressoHealthy = espressoHealthCheck(dc);
            // Check network connectivity
            boolean networkHealthy = networkHealthCheck(dc);
            
            if (kafkaHealthy && espressoHealthy && networkHealthy) {
                return DCStatus.HEALTHY;
            } else if (kafkaHealthy || espressoHealthy) {
                return DCStatus.DEGRADED;
            } else {
                return DCStatus.FAILED;
            }
        } catch (Exception e) {
            return DCStatus.FAILED;
        }
    }
}

Network Partition Resilience

Split-Brain Prevention

Prevent multiple Control Planes from accepting writes during network partition

// Distributed lock using Espresso for write leadership
public class WriteLeadershipController {
    
    private static final String LEADERSHIP_KEY = "feature_flags_write_leader";
    private static final Duration LEASE_TTL = Duration.ofMinutes(2);
    
    // Acquire write leadership with renewable lease
    public boolean acquireWriteLeadership(String datacenter) {
        try {
            LeadershipLease lease = LeadershipLease.builder()
                .datacenter(datacenter)
                .acquiredAt(System.currentTimeMillis())
                .expiresAt(System.currentTimeMillis() + LEASE_TTL.toMillis())
                .instanceId(getInstanceId())
                .build();
            
            // Atomic compare-and-swap operation
            boolean acquired = espressoClient.compareAndSet(
                LEADERSHIP_KEY, 
                null, // expected current value (null = no current leader)
                lease,
                LEASE_TTL
            );
            
            if (acquired) {
                scheduleLeaseRenewal();
                logger.info("Acquired write leadership for datacenter: {}", datacenter);
            }
            
            return acquired;
        } catch (Exception e) {
            logger.error("Failed to acquire write leadership", e);
            return false;
        }
    }
    
    // Renew lease every 30 seconds (before expiration)
    @Scheduled(fixedDelay = 30000)
    private void renewLease() {
        if (isWriteLeader()) {
            LeadershipLease currentLease = getCurrentLease();
            currentLease.setExpiresAt(System.currentTimeMillis() + LEASE_TTL.toMillis());
            
            espressoClient.put(LEADERSHIP_KEY, currentLease, LEASE_TTL);
        }
    }
}

  )
  
  // All regions should return same result
  const firstResult = results[0]
  const allConsistent = results.every(r => r.enabled === firstResult.enabled)
  
  if (!allConsistent) {
    throw new Error(`Flag drift detected: ${JSON.stringify(results)}`)
  }
  
  // Check evaluation latency
  const maxLatency = Math.max(...results.map(r => r.latencyMs))
  if (maxLatency > 2) {
    throw new Error(`Evaluation too slow: ${maxLatency}ms > 2ms SLO`)
  }
  
  return { consistent: true, maxLatency }
})

Error Budget & Burn Rate Alerts

// LinkedIn metrics for SLO tracking
class SLOMonitoring {
    
    // 99.999% availability SLO = 0.001% error budget
    private static final double ERROR_BUDGET = 0.00001; // 0.001%
    
    // Calculate error budget burn rate
    public double calculateBurnRate(Duration window) {
        long totalEvaluations = metricsClient.sum("flag_evaluations_total", window);
        long errorEvaluations = metricsClient.sum("flag_evaluations_errors", window);
        
        double errorRate = (double) errorEvaluations / totalEvaluations;
        double burnRate = errorRate / ERROR_BUDGET;
        
        return burnRate;
    }
    
    // Multi-window burn rate alerting (Google SRE book pattern)
    @Scheduled(fixedDelay = 60000) // every minute
    public void checkSLOBurnRate() {
        double oneHourBurn = calculateBurnRate(Duration.ofHours(1));
        double fiveMinBurn = calculateBurnRate(Duration.ofMinutes(5));
        
        // Alert if burning budget too fast
        if (oneHourBurn > 14.4 && fiveMinBurn > 14.4) {
            // 14.4x = exhaust monthly budget in 2 hours
            alertManager.page("SLO burn rate critical - budget exhausted in 2h");
        } else if (oneHourBurn > 6 && fiveMinBurn > 6) {
            // 6x = exhaust monthly budget in 5 hours  
            alertManager.warn("SLO burn rate high - budget exhausted in 5h");
        }
    }
}

Incident Response Playbooks

Runbook: High Flag Evaluation Latency

Alert: "Flag evaluation P99 >5ms for 5 minutes"

Investigation Steps:

  1. Check if specific service or global issue: grep "slow_flag_eval" service_logs
  2. Verify SDK cache health: Check cache hit ratio and memory usage
  3. Look for memory pressure: JVM GC logs, heap utilization
  4. Check for hash collision: Review user ID distribution in cohort assignment
  5. Escalate to on-call engineer if not resolved in 10 minutes

Mitigation:

  • Restart affected service instances to clear cache corruption
  • Temporarily disable new flag evaluations via circuit breaker
  • Force cache refresh from Memcache

Runbook: Flag Drift Detected

Alert: "Flag evaluation inconsistency >5% across DCs"

Investigation Steps:

  1. Identify affected datacenters and flag keys
  2. Check Kafka consumer lag: kafka-consumer-groups.sh --describe
  3. Verify network connectivity between DCs
  4. Review recent deployments or config changes
  5. Check Edge SDK version consistency across services

Mitigation:

  • Execute resync-all command: curl -X POST /admin/resync-all
  • Reset individual SDK caches via management endpoint
  • Use emergency broadcast channel to bypass Kafka
  • Temporary rollback to previous flag version if user impact

Runbook: Kill Switch Not Working

Alert: "Kill switch propagation >10s or failed"

Investigation Steps:

  1. Verify kill switch command reached all DC Kafka topics
  2. Check if Edge SDKs are processing kill switch events
  3. Look for SDK deployment issues or version mismatches
  4. Review network partitions or DC-level failures

Emergency Mitigation:

  • Direct Espresso update to force flag disable
  • Manual service restarts to force cache refresh
  • DNS-level traffic routing to bypass affected services
  • Engage incident commander for coordination

// Optimized flag evaluation with minimal allocations
public final class OptimizedFlagEvaluator {
    
    // Pre-computed hash cache to avoid repeated SHA-256
    private final ConcurrentHashMap<String, Long> hashCache = new ConcurrentHashMap<>(10000);
    
    // Thread-local objects to avoid allocation
    private static final ThreadLocal<MessageDigest> DIGEST = 
        ThreadLocal.withInitial(() -> {
            try {
                return MessageDigest.getInstance("SHA-256");
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        });
    
    // Hot path: zero allocation flag evaluation
    public boolean isEnabled(String flagKey, String userId) {
        FlagConfig flag = flagCache.get(flagKey); // O(1) hash lookup
        if (flag == null || !flag.enabled) {
            return false; // Fast path for disabled flags
        }
        
        // Use cached hash if available
        String cacheKey = flagKey + ":" + userId;
        Long cachedHash = hashCache.get(cacheKey);
        
        if (cachedHash == null) {
            // Compute hash and cache for future use
            cachedHash = computeUserHash(flagKey, userId);
            hashCache.put(cacheKey, cachedHash);
        }
        
        int bucket = (int) (Math.abs(cachedHash) % 100);
        return bucket < flag.rolloutPercentage;
    }
    
    // Batch evaluation for multiple flags (common pattern)
    public Map<String, Boolean> evaluateFlags(String userId, String... flagKeys) {
        Map<String, Boolean> results = new HashMap<>(flagKeys.length);
        
        for (String flagKey : flagKeys) {
            results.put(flagKey, isEnabled(flagKey, userId));
        }
        
        return results;
    }
    
    // Periodic cleanup of hash cache to prevent memory leaks
    @Scheduled(fixedDelay = 300000) // 5 minutes
    private void cleanupHashCache() {
        if (hashCache.size() > 50000) {
            // Remove oldest 25% of entries
            hashCache.clear(); // Simple strategy - could use LRU
        }
    }
}
