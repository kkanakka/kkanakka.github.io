The 6-Step SRE Design Methodology

Why This Framework Works

Google SRE interviews focus on reliability engineering rather than just distributed systems. This means every design decision must be justified by SLOs, failure modes, and operational concerns. The 6-step approach ensures you cover all critical aspects systematically.

1. Requirements → SLO Contract

"How reliable, fast, and scalable must this system be?"

  • Define availability targets (99.9%, 99.99%, 99.999%)
  • Set latency requirements (p50, p95, p99)
  • Establish throughput expectations (QPS, concurrent users)
  • Determine data consistency needs (strong, eventual, session)
  • Identify critical user journeys and their SLOs

2. SLO Contract → Baseline Design

"What minimal set of components meets those guarantees?"

  • Design the simplest architecture that meets SLOs
  • Choose appropriate databases and caching layers
  • Define API contracts and data models
  • Estimate capacity and resource requirements
  • Map components to specific SLO contributions

3. Baseline Design → Stressors

"What happens when scale ×10 or a dependency fails?"

  • Traffic spikes (10x normal load)
  • Database failures (primary, replica, partition)
  • Network issues (latency, packet loss, partitions)
  • Dependency outages (external APIs, services)
  • Resource exhaustion (CPU, memory, disk, network)

4. Stressors → Resilient Architecture

"Which reliability patterns mitigate failure modes?"

  • Replication (master-slave, master-master, sharding)
  • Circuit breakers and bulkheads
  • Queue-based decoupling and backpressure
  • Graceful degradation and fallbacks
  • Rate limiting and load shedding

5. Resilient Architecture → Observability & Ops

"How will we know it's healthy—and repair it safely?"

  • SLI metrics and monitoring dashboards
  • Error budgets and burn rate alerts
  • Distributed tracing and logging
  • Health checks and synthetic monitoring
  • Runbooks and incident response procedures

6. Observability & Ops → Trade-offs

"Where do we accept cost or complexity to meet SLOs?"

  • Performance vs consistency trade-offs
  • Cost of over-provisioning for reliability
  • Operational complexity vs automation
  • Development velocity vs safety
  • Regional deployment vs latency requirements

Design a health check service that can monitor 100,000 instances across multiple regions. Detect instance failure within 10 seconds with 99.99% detection reliability. Handle network partitions gracefully.

  • Network Partitions: Checker can't reach target but target is healthy

Majority Vote Pattern: 3 checkers per target, 2/3 must agree on failure

  • Multi-Region Checkers: Deploy in different AZs/regions
  • Exponential Backoff: Reduce check frequency for consistently failing targets
  • Circuit Breaker: Stop checking obviously dead instances
  • Consensus Algorithm: Require majority agreement for state changes

5. Observability & Ops

  • SLI: successful_checks / total_checks
  • Metrics: Check latency, success rate, consensus conflicts
  • Alerts: Detection SLO violation, checker health issues
  • Dashboards: Per-region health maps, check latency distribution

6. Trade-offs

✅ More Checkers
  • Lower false positive rate
  • Better network partition handling
  • Higher detection reliability
❌ Cost Implications
  • 3x compute cost for checker infrastructure
  • 3x network bandwidth usage
  • More complex consensus logic

Requirements

Design a URL shortener handling 100M URLs created daily, 1B redirects daily. 99.9% availability, p99 redirect latency < 100ms. Support custom aliases and analytics.

1. SLO Contract

MetricTargetMeasurement
Redirect Latencyp99 ≤ 100msFrom request to 302 response

Availability≥ 99.9%Successful requests / total
Durability≥ 99.99%URLs must not be lost

2. Baseline Design

Load Balancer
→
Web Servers
→
Cache Layer
→

  • Encoding: Base62 encoding for short URLs (a-z, A-Z, 0-9)
  • ID Generation: Auto-incrementing counter or UUID
  • Database: MySQL with URL mappings table
  • Cache: Redis for hot URL lookups

3. Stressors

  • Hot URLs: Viral links causing cache hot spots
  • Database Write Bottleneck: 100M creates/day = 1,157/sec
  • Cache Misses: New URLs not in cache cause DB load
  • Custom Alias Conflicts: Duplicate custom short codes

4. Resilient Architecture

Scaling Strategy

Read Replicas
+
Write Sharding
+
CDN Caching
  • Database Sharding: Shard by short_url hash for write distribution
  • Read Replicas: Separate read traffic from writes
  • CDN Layer: Cache 302 redirects at edge locations
  • Bloom Filter: Quick existence check before DB lookup
  • Rate Limiting: Prevent abuse and ensure fair usage

5. Observability & Ops

  • SLI: redirect_success_rate, creation_success_rate
  • Metrics: Cache hit ratio, DB query latency, URL creation rate
  • Alerts: High error rate, cache miss spikes, DB connection issues
  • Analytics: Click tracking, geographic distribution, referrer data

6. Trade-offs

✅ CDN Benefits
  • Sub-50ms redirect latency globally
  • Reduced backend load (80%+ cache hit)
  • DDoS protection
❌ CDN Challenges
  • Analytics data collection complexity
  • Cache invalidation delays
  • CDN costs for high traffic

  • Complex cross-region synchronization
  • Reduced availability during partitions

  • Storage: Time-series database (InfluxDB/TimescaleDB)
  • Querying: PromQL-style query language
  • Alerting: Rule evaluation engine with notification routing

3. Stressors

  • Metric Explosion: High-cardinality labels causing storage issues
  • Query Overload: Complex dashboard queries impacting ingestion
  • Storage Growth: Retention policies vs query performance
  • Alert Storms: Cascading failures triggering thousands of alerts

4. Resilient Architecture

Federated Architecture

Shard A
↘
Federation
↙
Shard C
  • Horizontal Sharding: Distribute metrics by service/region
  • Downsampling: Reduce resolution for older data
  • Query Federation: Cross-shard queries with result aggregation
  • Alert Deduplication: Group related alerts to prevent storms
  • Circuit Breaker: Disable expensive queries during overload

5. Observability & Ops

  • SLI: successful_ingestions/total, query_success_rate
  • Meta-monitoring: Monitor the monitoring system itself
  • Capacity Planning: Storage growth rate, query load trends
  • Performance Optimization: Index usage, query plan analysis

6. Trade-offs

✅ High Retention
  • Long-term trend analysis
  • Capacity planning data
  • Incident forensics
❌ Storage Costs
  • Exponential storage growth
  • Query performance degradation
  • Backup and recovery complexity

  • Geographic Distribution: 200+ Points of Presence globally
  • Cache Hierarchy: Edge → Regional → Origin
  • Content Routing: Anycast DNS for nearest POP selection
  • Storage Tiering: SSD for hot content, HDD for warm content

3. Stressors

  • Flash Crowds: Viral content causing cache storms
  • Large File Delivery: Video files exhausting bandwidth
  • Cache Invalidation: Stale content after origin updates
  • Geographic Hotspots: Regional traffic concentration

4. Resilient Architecture

Multi-Tier Caching Strategy

→
L2 Cache (HDD)
→

  • Consistent Hashing: Distribute content across cache nodes
  • Adaptive Bitrate: Multiple quality levels for video content
  • Prefetching: Predictive content loading based on patterns
  • Load Balancing: Weighted round-robin with health checks
  • DDoS Protection: Rate limiting and traffic filtering

5. Observability & Ops

  • SLI: cache_hit_ratio, origin_latency, edge_availability
  • Real-time Monitoring: Traffic patterns, cache performance
  • Geographic Analytics: Per-region latency and throughput
  • Content Analytics: Popular content identification for prefetching

6. Trade-offs

✅ More Edge Locations
  • Lower latency globally
  • Better user experience
  • Reduced origin load
❌ Operational Complexity
  • Higher infrastructure costs
  • Complex cache invalidation
  • Content synchronization challenges
