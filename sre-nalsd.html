Scenarios 1-5: Network, CDN, DNS

S1 — Regional Latency Spike (South America 500ms Slower)

Symptom

Photo-sharing uploads from SA consistently 500ms slower than other regions. No errors, just slowness. Rest of world normal.

Diagnostic Path

Layers to check: Client → DNS (correct CDN edge?) → CDN (São Paulo PoP hit or routed to Miami?) → LB (BGP routing asymmetry?) → App (SA cluster healthy?) → DB (writes routed cross-continent?)

Key question: Is 500ms constant (misrouting) or intermittent (congested ISP link)?

Root Cause (Example)

CDN config change invalidated São Paulo upload caching tier → 40% of requests proxied back to US-East → +500ms RTT cross-continent.

Mitigation

Now: Route traffic to nearest working CDN edges. Roll back config. Long-term: Synthetic per-region latency checks. Alerts on CDN config drift. Fail-safe fallback so CDN never reverts to US-East for uploads.

Senior Signal

Strong: diagnoses CDN misconfig + proposes rollback. Exceptional: adds monitoring, DNS TTL tradeoffs, automation.

Pitfalls: Blaming app/DB. Not asking scope questions. Only proposing fixes, not mitigations (interviewers want restore first).

S2 — Load Balancer Connection Storm (SYN Flood / Flash Crowd)

Symptom

LB connection queues at max. Spike in half-open TCP connections (SYN_RECV). Users see intermittent timeouts. App servers underutilized (CPU fine).

Diagnostic Path

netstat -s | grep SYN    # SYN backlog
cat /proc/net/nf_conntrack | wc -l  # conntrack table
ethtool -S eth0 | grep drop

Check: SYN backlog → conntrack → LB timeout config → app rejecting connections → DDoS vs flash crowd (handshake completion ratio).

Root Cause

SYN flood: 80% of connections never complete handshake. Source IPs randomized, no valid ACKs. LB TCP backlog saturated.

Mitigation

Now: Enable SYN cookies (net.ipv4.tcp_syncookies=1). Rate-limit suspicious IPs. Autoscale LBs. Long-term: DDoS mitigation at edge. Tune tcp_max_syn_backlog, reduce tcp_synack_retries. Scrubbing layer before LBs.

Senior Signal

Strong: quickly identifies LB/kernel-level, not app. Talks SYN backlog, conntrack. Exceptional: explains SYN flood vs legit flash crowd. Layered defense: kernel tuning, scrubbing, monitoring.

S3 — TLS Certificate Rotation Failure

Symptom

TLS handshake latency +2s. Some regions: complete failures. Logs: "certificate expired." Service itself healthy.

Root Cause

Cert rotation job failed on ~20% of global LBs (still serving expired cert). Where updated, cipher preference change caused negotiation retries → latency spike.

Mitigation

Now: Force cert rollout to all LBs. Revert cipher config. Long-term: Fully automate cert rotation (ACME). Synthetic TLS handshake tests per region/browser. Pre-rotation overlap. Auto-rollback on latency spike post-rotation.

S4 — DNS Resolution Degradation (30% NXDOMAIN)

Symptom

~30% of global users: "site not reachable." Others fine. Pattern: NXDOMAIN or SERVFAIL. Not uniform across ISPs.

Root Cause

Authoritative DNS cluster has 4 servers. Zone propagation automation bug → 2 servers correctly updated, 2 still serve old data. Recursive resolvers hit "good" or "stale" servers randomly.

Mitigation

Now: Force zone reload on all auth servers. Lower TTL temporarily. Long-term: Validate all auth servers before completing push. Synthetic tests from multiple resolvers. Dual-push with pre-validation. Alert on zone drift.

Senior Signal

Exceptional: explains why some users fail (cache + server inconsistency). Mentions propagation validation, drift detection, toil reduction.

S5 — CDN Cache Invalidation Delay

Symptom

New frontend release: blank screens, misaligned buttons. 404s for versioned assets. Some users old assets, others new. Backend healthy.

Root Cause

Cache invalidation API timeout → ~15% of PoPs never got purge. Those serve old JS → broken UX.

Mitigation

Now: Manual global purge. Cache-busting query params. Long-term: Content-hash versioning (app.<sha>.js). Post-deploy synthetic PoP checks. Retry purge failures. Staged PoP rollout.

Scenarios 6-9: Database, Memory, Network Internals

S6 — Database Replica Lag in Multi-Region Service

Symptom

Users in EU + SA see stale data after posting. Replication lag 30-60s behind primary (US-East). App metrics healthy.

Root Cause

Write traffic spiked 3x (new feature). Replicas on slower disks → WAL replay can't keep up. App still routing reads to lagging replicas.

Mitigation

Now: Route fresh reads to primary. Degrade gracefully for non-critical features. Long-term: Upgrade replica I/O. Lag-aware read-routing (if lag > 5s → failover to primary). Semi-sync replication. Regional primaries.

Senior Signal

Exceptional: walks through binlog/WAL replay mechanics. Proposes lag-threshold routing. Balances consistency vs performance tradeoffs.

S7 — Memory Leak in Go Service (Heap Bloat)

Symptom

Go API service: rising memory over days, hits OOM → restart every ~48h. GC spikes before crash. CPU normal. Latency climbs before OOM.

Diagnostic

go tool pprof http://pod:6060/debug/pprof/heap
pprof> top
3.07GB 80% .../search/cache.go:45

Root Cause

Request logging buffer never cleared after last release. Memory accumulates until OOM.

Mitigation

Now: Restart every 12h. Flush buffers. Increase memory limit. Long-term: Fix buffer rotation code. Unit tests for memory under load. Heap monitoring + SLO alert on memory growth rate.

Senior Signal

Juniors: "Add more memory." Strong: "pprof the leak." Senior: "Every OOM is user-visible availability hit. New cache features must pass leak-test before rollout."

S8 — Packet Drops Under Kernel Resource Contention

Symptom

Front-end proxies: sporadic 5xx during traffic spikes. CPU 40-55% but network RX drops climbing. SYN_RECV and retransmits jump.

Diagnostic

ethtool -S ens5          # rx_no_buffer, rx_dropped
cat /proc/net/softnet_stat  # dropped, squeezed per CPU
cat /proc/interrupts        # All NIC IRQs on CPU0!
ethtool -g ens5             # RX ring size (too small?)

Root Cause

irqbalance disabled after kernel upgrade → all NIC IRQs land on CPU0 → single-core choke. RX ring at conservative default. Packet path: NIC → ring → NAPI → softirq → qdisc → socket. Softirq saturated on CPU0.

Mitigation

Now: Spread IRQ affinity. Enable RSS/RPS/XPS. Increase RX ring + netdev_max_backlog. Long-term: NUMA locality enforcement. fq_codel qdisc. eBPF observability. Golden image validation for RSS/IRQ post-upgrade.

Senior Signal

Exceptional: explains packet path NIC→ring→NAPI→softirq→qdisc→socket. Brings in NUMA, fq_codel, eBPF. Proposes golden image CI.

S9 — Cross-Region Data Transfer Bottleneck (BDP)

Symptom

Single TCP flows cap ~120-200 Mbps on 1-10 Gbps links. RTT 120-220ms. Low loss (0.05-0.2%). Intra-region transfers saturate links fine.

Diagnostic

BDP = bandwidth × RTT
1 Gbps × 200ms = 25 MB in-flight required
If rwin/cwnd < 25 MB → never hit line rate

sysctl net.ipv4.tcp_rmem   # Check max receive buffer
sysctl net.ipv4.tcp_congestion_control  # CUBIC? BBR?

Root Cause

tcp_rmem max = 4MB (far below BDP of 25MB). CUBIC + tiny losses depress cwnd further. Per-flow policer at ~200 Mbps. Single stream = capped.

Mitigation

Now: Parallelize streams (8-16 TCP). Increase rmem/wmem to 32MB. Switch to BBR. Enable jumbo MTU. Long-term: Negotiate per-flow policer raise. Self-tuning transfer service. Goodput-centric SLOs.

Senior Signal

Exceptional: does BDP math. Calls out per-flow shaping. Proposes auto-tuning transfer service with SLOs for goodput (not link rate).

Scenarios 10-13: Control Plane, Time, Storage, Discovery

S10 — Control-Plane Failure Bleeding into Data Plane

Symptom

Config/feature-flag service degraded (quorum lost). Healthy pods deny requests: "policy fetch failed." New pods fail to start (init waiting on config). Backends + DBs healthy.

Root Cause

Client-lib update changed config fetch from background refresh to on-demand per-request validation for "high-risk flags." Fail-closed policy. TTL=30s with hard revalidation. Control-plane hiccup → cascaded into data plane.

Mitigation

Now: Force fail-open. Raise cache TTLs. Enable serve-stale. Un-gate init. Circuit break + backoff control-plane clients. Long-term: Decouple data plane from control plane by design. Out-of-band background refresh with sticky last-known-good (LKG). Signed snapshots via CDN. Game days: disable CP and prove DP meets SLOs.

Senior Signal

Exceptional: principled resilience model (LKG + signed snapshots + strict retry budgets). Differentiates security-critical vs non-critical. Adds SLOs for degraded mode.

S11 — Time Skew & NTP Failure

Symptom

Auth failures (tokens "expired"/"not yet valid"). TLS handshakes fail. Crons run twice or skip. Logs out of order. Negative latencies on dashboards.

Root Cause

EU NTP cluster lost GPS refclock (hardware fault). Drifted +4 minutes over 24h. Auth services rejected tokens from US ("not valid yet"). TLS certs looked expired. Logs showed negative durations.

Mitigation

Now: Resync against healthy external NTP. Widen time tolerance for auth/TLS. Gradual slew (not step). Long-term: Redundant NTP (GPS + multiple upstreams). Chrony/TrueTime-style. Use CLOCK_MONOTONIC for durations. Bounded skew for token validation. Game days: simulate NTP loss.

Senior Signal

Exceptional: discusses monotonic vs wall clocks. Mentions TrueTime. Frames time as a distributed systems dependency.

S12 — Disk I/O Saturation (IOPS vs Throughput)

Symptom

Latency spikes on reads/writes. IOPS pegged but throughput well below capacity (150 MB/s on 500 MB/s disks). Errors only under spiky workloads.

The Math

Disks: 10K IOPS quota, 500 MB/s throughput
Workload: 4 KB random writes
Max throughput = 10K IOPS × 4 KB = 40 MB/s
(NOT 500 MB/s — IOPS is the bottleneck, not throughput!)

Root Cause

Small 4KB random writes from DB logs. Hit per-volume IOPS cap. Queue depth grows → await latency spikes → replicas marked "unhealthy."

Mitigation

Now: Stripe across volumes. Cache hot reads. Throttle background jobs. Long-term: NVMe/provisioned IOPS. Redesign for sequential batching. Tiered storage. I/O admission control.

Senior Signal

Exceptional: shows math (10K IOPS × 4KB = 40 MB/s). Talks retry storms and admission control. Proposes tiered storage + log-structured designs.

S13 — Service Discovery Failure (CoreDNS/etcd/Mesh Crash)

Symptom

Hundreds of services return 503. Logs: "host not found" or "no healthy backends." Backends themselves healthy. Services can't locate each other.

Root Cause

K8s upgrade deployed new CoreDNS config with bug → crashloop on invalid zone file. All DNS lookups for svc.cluster.local fail. Services with cached IPs worked until TTL expired (~30s).

Mitigation

Now: Rollback CoreDNS config. Extend DNS TTL. Inject static DNS for critical services. Long-term: Decouple discovery from main cluster. Redundant layers (DNS fallback, LKG in sidecars). Canary CoreDNS changes. Game days: simulate discovery outage.

Scenarios 14-18: Rate Limiting, Queues, Observability, Quotas

S14 — Rate-Limiter Gone Rogue

Symptom

Surge of 429s. Traffic dropped 40%. Backends healthy. Rate-limiter update rolled out earlier.

Root Cause

Config parsed "1000 req/s per tenant" as "1000 req/minute" (off by 60x). All tenants exceeded quota instantly → majority of requests rejected.

Mitigation

Now: Roll back config. Temporarily fail-open if rollback blocked. Long-term: Config validation with unit checks. Canary with synthetic traffic. Fail-open for limiter backend outages. Per-tenant fairness. Alerts: if rejection >5% globally, block rollout.

Senior Signal

Exceptional: explains limiter algorithms (token/leaky bucket). Proposes synthetic pre-flight validation. SLOs for rate-limiter (<1% unexpected 429s). Security vs availability trade-off.

S15 — Queue Backpressure Cascade

Symptom

Multi-tier: API → Orchestrator → Worker → DB. p95/p99 explodes. Retry rates climb. Queue depths rise. OOMs in workers. System self-amplifying.

Root Cause

New feature doubled writes to hot DB partition → worker throughput drops 50% → Kafka lag grows → API retries 3x with no jitter (synchronized storm) → producers buffer in RAM → OOM → replayed messages → amplifying loop.

Mitigation

Now: Trip circuit breakers on slow downstream. Reduce retry count, enforce backoff+jitter. Per-tenant rate limits. Drain: pause feature causing hot-spot. Cap producer buffers. Long-term: Idempotent handlers + dedup keys. Hedged requests with budgets. Shard hot keys. Queue SLOs for lag. Auto-circuit on threshold. Game days.

Senior Signal

Exceptional: systemic blueprint — idempotency, hedged requests, budgeted concurrency, priority queues. Control-theory: negative feedback loops to dampen oscillations.

S16 — Log Pipeline Collapse (Observability Outage)

Symptom

Service serves normally — no user impact. But: central log pipeline hours behind. Dashboards show flatline/old data. Alerts: "no data received." On-call engineers blind.

Root Cause

New release added large debug field → log size 5x → Kafka partition disk I/O limit → consumer lag grew from minutes to hours. Metrics pipeline also slowed (shared brokers).

Mitigation

Now: Drop non-critical log fields. Throttle debug logging. Increase Kafka partitions. Create side-channel lightweight synthetic metrics. Long-term: Log budgets per service. Separate serving metrics from debug logs. Canary log schema changes. SLOs for observability ("95% logs ingested <5m"). Graceful shedding (drop debug first).

Senior Signal

Exceptional: treats observability as production-critical system with its own SLOs. Prioritized: serving metrics > logs > debug. Side-channel synthetic probes when blind.

S17 — Cloud Resource Quota Exhaustion

Symptom

Traffic surge → autoscaler kicks in → new VMs/pods fail: "Quota exceeded." API Gateway 503s. Some regions scale fine, others flatline. Existing instances healthy but overloaded.

Root Cause

GCP CPU quota per project per region = 2,000. Current usage = 2,000. Autoscaler requested 200 more → API error. No quota alerts had been set up.

Mitigation

Now: Reroute traffic to other regions. Emergency quota raise request. Throttle traffic. Scale down batch jobs to free quota. Long-term: Quota-aware autoscaler. Pre-provisioned 20% headroom. Quota monitoring + alerting. Multi-project pooling. Synthetic drills.

S18 — API Gateway Token Bucket Bug

Symptom

Surge of 503s and 429s. Even low-traffic clients rejected. Backends healthy. New Gateway release rolled out.

Root Cause

Gateway config parser: refill rate "1000 tokens per second" interpreted as "1000 per minute." All clients exhausted quota instantly → 70% of traffic rejected.

Mitigation

Now: Roll back release. Fail-open if rollback blocked. Long-term: Config schema validation with explicit units. Canary with synthetic load. Per-tenant token buckets (not global). Fail-open for availability-critical, fail-closed only for security. Chaos drills for quota backend outage.

Self-Practice Template
BLANK NALSD TEMPLATE (10-12 min per scenario):

Symptom:         (Write the user-facing symptom)
Layers to Check: Client → Network → Infra → Service → Data → OS/Kernel
Hypotheses:      H1: ... H2: ... H3: ...
Stabilization:   (What you'd do right now as on-call)
Prevention:      (Design change, automation, monitoring, chaos test)
Reflection:      What did you prioritize first, and why?
                 What trade-offs did you face?
                 Did you propose both now-fix AND long-term prevention?

MINI DRILL (one-liner):
Symptom → Hypotheses → First Check → Mitigation → Long-term Fix

Example:
  Symptom: Some users see stale CSS after deploy.
  Hypotheses: CDN invalidation delay, wrong cache headers.
  First Check: Inspect CDN edge vs origin headers.
  Mitigation: Force purge, serve cache-busted URLs.
  Long-term: Content-hash filenames, canary purge test.
