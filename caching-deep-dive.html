2. Cache Miss — What Actually Happens

Cache miss is always the same three steps — detect miss, go to DB, populate cache — but the cause differs.

Universal Cache Miss Flow

App request
➜
GET x
➜
Cache: MISS
➜
DB fetch
➜
SET x
➜
Cache warm

Happens once per key. Normal and expected.

🥶 Cold Miss

Key never loaded before

  • Cache restart/empty
  • First access to new data
  • Unavoidable at startup
Normal and expected. Every key experiences this once.

🗑️ Capacity/Eviction Miss

Key was evicted (LRU/LFU/TTL)

  • Cache too small for working set
  • TTL expired
  • Random eviction in Redis
Tune maxmemory and eviction policy. Monitor hit rate.

🗂️ Invalidation Miss

Key was deleted on DB write

  • Delete on write (cache-aside)
  • Update on write (write-through)
  • TTL-based invalidation
By design. Strategies: delete vs update vs TTL.
Cache Stampede Risk: Many requests miss at once → all hit DB simultaneously.
Fix: Mutex lock, probabilistic early expiry (PER), or request coalescing (only 1 thread fetches, rest wait).

Cache Stampede Problem

Problem: Cache stampede: many requests miss at once → all hit DB simultaneously

Cache: MISS
→
100x DB requests

Solutions:

  • Mutex lock: Only 1 thread fetches, others wait
  • Probabilistic early expiry (PER): Refresh before TTL
  • Request coalescing: Combine identical requests

Production Standard: RDB + AOF Together

RDB for fast restart, AOF for durability. Use both together:
• 0-1s data loss (AOF fsync frequency)
• Full data restored (RDB snapshot + AOF replay)
• Reasonable restart time (RDB loads fast, AOF fills gaps)

The Cache Has Zero DB Knowledge — Here's What Actually Happens

❌ Common Myth

App
→
🏪 Cache
→
DB

❌ "Cache automatically fetches from DB"
❌ "This is something cache libraries do on their own."

✅ Reality

App code (in controller):
val = redis.get(key)
if val is None:
  val = db.find(id)
  redis.set(key, val)
return val

App fetches from DB on cache miss.
Framework intercepts call in proxy layer.
Query lib handles it transparently.

Three Implementation Approaches — Same Result, Different Writers

1. App Cache-aside (manual)

You write the DB query.
You call cache.set() to populate.
Framework: your own

2. Cache client with loader

Framework intercepts call.
Detects cache miss, calls your loader func.
Returns result to app your_loader = db.find(id)

3. Proxy (transparent)

Rank App sees proxy as normal.
Proxy checks cache before DB.
Returns result to app Even the DB Proxy.

In all three cases, the cache only speaks GET/SET with your app. The loader function — written by you or your framework — decides the protocol: MySQL, gRPC, HTTP. Who actually opens the DB query?

Key Insight: Memcached and Redis are caches — they have no idea where your data originally came from.
Couchbase is a primary database that happens to keep everything in memory first.

Choose based on: Do you need the cache to survive restarts? Do you need rich data types? Is this your only datastore?
