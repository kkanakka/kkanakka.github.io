1. Why Two Caches? The Bootstrap Problem

Many systems use two caching layers: a local in-process cache (populated via Kafka deltas) and a shared Memcache/Redis cluster. The question is why both are needed.

❌ Without Memcache

Step 1: New service starts
Step 2: Empty local cache — nothing to serve
Step 3: Wait for Kafka stream to deliver flags
Step 4: Gradually receive flags (minutes)
Result: Cold start. Minutes of degraded service.

✅ With Memcache

Step 1: New service starts
Step 2: Load full snapshot from Memcache (~100ms)
Step 3: Already fully warm — start serving
Step 4: Tail Kafka for real-time deltas only
Result: Instant warm start. No cold window.
Cache Type Update Frequency Update Method Access Pattern
Local cache Real-time Kafka deltas Every request (millions/sec)
Memcache / Redis Every 30s Control Plane batch write Only at startup / warm-up
Key insight: These two caches are not redundant — they serve completely different jobs. The local cache handles millions of requests per second. Memcache exists only to eliminate the cold-start window when a new instance boots.

3. Cache Miss — What Actually Happens

Cache miss is always the same three steps — detect miss, go to DB, populate cache — but the cause differs.

Universal Cache Miss Flow

App request
➜
GET x
➜
Cache: MISS
➜
DB fetch
➜
SET x
➜
Cache warm

Happens once per key. Normal and expected.

Miss Type Cause How Often Prevention
Cold miss Key was never loaded into cache Once per key, on first access Eager pre-warm at startup
Capacity/eviction miss Key was evicted by LRU, LFU, or TTL expiry Ongoing if working set > cache size Increase maxmemory; tune eviction policy
Invalidation miss Key was deleted when DB was updated Intentional — by design Minimise by using write-through or short TTL

🥶 Cold Miss

Key never loaded before

  • Cache restart/empty
  • First access to new data
  • Unavoidable at startup
Once per key, on first access. Eager pre-warm at startup.

🗑️ Capacity/Eviction Miss

Key was evicted (LRU/LFU/TTL)

  • Cache too small for working set
  • TTL expired
  • Random eviction in Redis
Ongoing if working set > cache size. Increase maxmemory; tune eviction policy.

🗂️ Invalidation Miss

Key was deleted on DB write

  • Delete on write (cache-aside)
  • Update on write (write-through)
  • TTL-based invalidation
Intentional — by design. Minimise by using write-through or short TTL.
Cache stampede (thundering herd)
When many requests miss at the same time — e.g. after a restart or a mass expiry — they all hit the DB simultaneously, potentially overwhelming it. Three common fixes:

Cache Stampede Solutions

🔒 Mutex Lock

Only one request fetches from DB; the rest wait and then read the now-warm key.

⏰ Probabilistic Early Expiry (PER)

Re-compute a cache entry slightly before it expires, rather than waiting for a hard miss.

🔄 Request Coalescing

Deduplicate in-flight requests for the same key so only one goes to the DB.

The Cache Has Zero DB Knowledge — Here's What Actually Happens

❌ Common Myth

App
→
🏪 Cache
→
DB

❌ "Cache automatically fetches from DB"
❌ "This is something cache libraries do on their own."

✅ Reality

App code (in controller):
val = redis.get(key)
if val is None:
  val = db.find(id)
  redis.set(key, val)
return val

App fetches from DB on cache miss.
Framework intercepts call in proxy layer.
Query lib handles it transparently.

Three Implementation Approaches — Same Result, Different Writers

1. App Cache-aside (manual)

You write the DB query.
You call cache.set() to populate.
Framework: your own

2. Cache client with loader

Framework intercepts call.
Detects cache miss, calls your loader func.
Returns result to app your_loader = db.find(id)

3. Proxy (transparent)

Rank App sees proxy as normal.
Proxy checks cache before DB.
Returns result to app Even the DB Proxy.

In all three cases, the cache only speaks GET/SET with your app. The loader function — written by you or your framework — decides the protocol: MySQL, gRPC, HTTP. Who actually opens the DB query?

7. How Cache Knows to Pull from DB

Spoiler: it doesn't — the app does. The cache is a passive key-value store.

❌ The Myth

"Cache somehow detects a miss and automatically queries MySQL."

App
→
🏪 Cache
→
DB

✅ The Reality

Cache returns NIL. Your app code (or your framework's cache library) detects the miss, runs a loader function you wrote, fetches from DB, then calls redis.SET to populate the cache.

Approach Who writes the loader How it looks
Manual (cache-aside) You — in application code
val = redis.get(key)
if val is None:
  val = db.query("SELECT ...")
  redis.setex(key, ttl, val)
return val
Framework (@Cacheable) The framework — intercepts method calls
@Cacheable("users")
User getUser(Long id) {
  // runs ONLY on cache miss
  return db.findById(id);
}
Proxy / sidecar The proxy — sits between app and DB App sends SQL to proxy. Proxy checks cache → miss → queries DB → populates cache → returns result. App sees nothing special.

8. Protocols Used on Cache Miss

The loader function in your app code uses whatever protocol the target database speaks. The cache is not involved — it only ever sees GET and SET on its own protocol.

Database Protocol / driver Port Example call
MySQL / MariaDB MySQL wire protocol TCP 3306 SELECT * FROM users WHERE id = ?
PostgreSQL PG frontend/backend protocol TCP 5432 SELECT * FROM orders WHERE user_id = $1
Microservice (gRPC) HTTP/2 + Protobuf TCP 443 UserService.GetUser({user_id: "123"})
REST API / 3rd party HTTP/1.1 or HTTP/2 TCP 443 GET /api/users/123 Authorization: Bearer ...
MongoDB MongoDB wire protocol TCP 27017 db.users.findOne({_id: ObjectId(...)})
DynamoDB HTTPS + AWS Signature V4 TCP 443 dynamodb.get_item(Key={"id": "123"})
Cassandra CQL binary protocol TCP 9042 SELECT * FROM users WHERE id = ?
Espresso (LinkedIn) HTTP REST + Avro TCP 80/443 GET /espresso/profiles/{memberId}

After the loader returns: regardless of which DB and protocol was used, your code always calls the same cache command:

redis.set(key, serialized_value, ex=TTL_SECONDS)
# or: memcache.set(key, value, time=TTL_SECONDS)

The cache only speaks its own protocol (RESP for Redis on TCP 6379, Memcached binary/text on TCP 11211). Everything else is your app's concern.

9. LinkedIn: Espresso + Couchbase

A real-world case study in why you need both — LinkedIn reaches over 4.8 million profile reads per second

Layer System Role Protocol
Application Profile backend service Business logic Internal RPC
Cache (L1 hot keys) Off-heap cache (OHC) on Espresso router In-process hot key cache In-memory
Cache (L2 full dataset) Couchbase Full profile dataset cached per DC Memcached protocol
Primary database Espresso (LinkedIn-internal) Source of truth — disk-backed MySQL/InnoDB HTTP + Avro
Change stream Kafka / Brooklin Propagates every write to all consumers Kafka protocol

Why Couchbase could not replace Espresso

Dimension Espresso Couchbase
Availability LinkedIn internal — not public Open source + Enterprise
CAP choice CP — consistency + partition tolerance. Built on MySQL/InnoDB. AP by default — eventual consistency. Strong consistency costs latency.
Change capture First-class citizen. Every write produces an ordered SCN event via MySQL binlog → Kafka. 30+ downstream systems depend on this. DCP (Database Change Protocol) exists but is not integrated into the broader data ecosystem as a first-class feature.
Schema enforcement Avro schemas strictly enforced. All consumers in the data ecosystem (search, ML, Hadoop) agree on document structure. Schemaless JSON by default. JSON Schema support is newer and optional.
Storage engine MySQL / InnoDB — 30+ years of production trust. Binlog is the foundation of the change stream. Couchstore / Magma — proprietary, good but not as battle-tested.
When chosen Built in 2011, production June 2012. No existing system met LinkedIn's needs at that scale. Couchbase 1.0 also 2011 — immature, unproven at LinkedIn scale at that time.

Key takeaway: Couchbase didn't replace Espresso — it became a tenant of the Espresso ecosystem, fed by the same Kafka change stream it could never have produced itself. The right question is never 'which cache system?' but 'what role does each layer play, and what are the tradeoffs?'
