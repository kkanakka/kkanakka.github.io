Page 1 — System Overview & Security Philosophy

What This System Is

A "Production Secrets Rotation Service" that securely manages and rotates credentials (API keys, TLS certs, DB passwords, OAuth tokens) across LinkedIn's infrastructure. The system must minimize downtime during rotation and prevent configuration drift between services while maintaining zero-trust security principles.

Security Philosophy: Zero-Trust with Graceful Handoff

Core Security Principles

  • Dual-Key Rotation Pattern — Maintain old_key + new_key overlap window (15 min) for zero-downtime handoff
  • Least Privilege Access — Services only access secrets they need, with time-limited tokens
  • Immutable Audit Trail — Every secret access logged with cryptographic integrity
  • Fail-Safe Rollback — Always retain previous valid version for emergency recovery
  • Encryption at Rest & Transit — All secrets encrypted with HSM-backed keys
  • Versioned Secrets Schema — Consumers specify secret_id@version for controlled access

Five Core Components

  1. Secrets Store — Encrypted storage with HSM backing (LinkedIn Vault + Espresso)
  2. Rotation Controller — Policy-driven scheduler with automated rotation logic
  3. Distribution Plane — Secure propagation via Kafka with TLS + mTLS auth
  4. Consumer Agent — Sidecar daemon that fetches, decrypts, and hot-reloads secrets
  5. Audit Logger — Immutable compliance log with hash-chaining integrity

Reliability Stressors (What Can Break)

Critical Failure Modes

  • Partial Rollout: Some services pick up new secrets; others still use old ones → authentication split-brain
  • Configuration Drift: Service instance caches outdated keys → intermittent auth failures
  • HSM Latency Spikes: Decrypt operations stall rotation controller → delayed propagation
  • Network Partitions: Region A rotates, Region B misses update → cross-region inconsistency
  • Rotation Policy Error: Bad policy (e.g., "rotate every 10 seconds") → self-inflicted DDoS
  • Blast Radius: Single rotation bug breaks authentication for thousands of services

Production Design Patterns

Advanced Reliability Patterns

  • Dual-Key Rotation Pattern: 15-minute overlap window where both old/new keys are valid
  • Pull + Push Hybrid: Push via Kafka + periodic pull for self-healing consistency
  • Versioned Secrets Schema: db_password@v1234 for precise version control
  • Fail-Safe Rollback: One-click revert to previous_valid_version
  • Dead Letter Queue: Failed rotation attempts for manual investigation
  • Multi-Region Replicas: Secrets Store replicated across 4 LinkedIn datacenters
  • Automatic Revocation Alert: If rotation exceeds SLO window, auto-revoke old secrets

Consumer Agent (Sidecar Pattern)

Zero-Downtime Secret Injection: Hot-reload without service restart

// LinkedIn Consumer Agent
public class SecretConsumerAgent {
    private final ConcurrentHashMap<String, SecretVersion> secretCache;
    private final VaultClient vaultClient;
    private final KafkaConsumer<String, SecretEvent> consumer;
    
    // Hot-reload secret without service restart
    public void rotateSecret(SecretEvent event) {
        try {
            // Download new secret version
            SecretVersion newSecret = vaultClient.getSecret(
                event.getSecretId(), 
                event.getNewVersion()
            );
            
            // Decrypt with service's private key
            String decryptedSecret = decrypt(newSecret.getEncryptedValue());
            
            // Write to secure file location
            writeSecretToFile(event.getSecretId(), decryptedSecret);
            
            // Signal service to reload (SIGHUP or API call)
            signalServiceReload(event.getSecretId());
            
            // Report success to audit log
            auditLogger.info("Secret rotated successfully: {}", event.getSecretId());
            
        } catch (Exception e) {
            // Rollback to previous version if new rotation fails
            rollbackToPreviousVersion(event.getSecretId());
            alertManager.page("Secret rotation failed: " + event.getSecretId());
        }
    }
    
    // Periodic drift detection
    @Scheduled(fixedDelay = 60000) // every minute
    public void checkForDrift() {
        for (String secretId : secretCache.keySet()) {
            SecretVersion latest = vaultClient.getLatestVersion(secretId);
            SecretVersion local = secretCache.get(secretId);
            
            if (!latest.getVersion().equals(local.getVersion())) {
                logger.warn("Drift detected for {}: local={}, remote={}", 
                           secretId, local.getVersion(), latest.getVersion());
                // Self-heal by pulling latest version
                rotateSecret(createSyncEvent(secretId, latest));
            }
        }
    }
}

Page 3 — Implementation Details & Dual-Key Rotation

Dual-Key Rotation Pattern

Zero-Downtime Handoff Implementation

The core innovation: maintain both old and new secrets during a 15-minute overlap window

// Dual-key rotation state machine
public enum RotationState {
    STABLE,           // Only old_key active
    ROTATION_START,   // Both old_key + new_key active  
    OVERLAP_WINDOW,   // 15-minute grace period
    ROTATION_COMPLETE // Only new_key active
}

public class DualKeyRotationManager {
    
    // Execute zero-downtime rotation
    public void executeRotation(String secretId) {
        SecretMetadata secret = getSecretMetadata(secretId);
        
        // Phase 1: Generate new secret version
        String newVersion = generateNewSecret(secretId);
        secret.setState(ROTATION_START);
        secret.setNewKey(newVersion);
        secret.setOverlapEnd(Instant.now().plus(Duration.ofMinutes(15)));
        
        // Phase 2: Distribute new version (both keys valid)
        distributeSecretUpdate(secretId, secret);
        
        // Phase 3: Wait for all consumers to acknowledge
        waitForConsumerAcknowledgments(secretId, Duration.ofMinutes(5));
        
        // Phase 4: Validate no authentication errors
        if (validateMetricGates(secretId)) {
            // Phase 5: Complete rotation (revoke old key)
            secret.setState(ROTATION_COMPLETE);
            secret.setOldKey(null);
            distributeSecretUpdate(secretId, secret);
        } else {
            // Rollback: revert to old key only
            rollbackRotation(secretId);
        }
    }
    
    // Consumer validation logic
    public boolean authenticateWithSecret(String secretId, String providedSecret) {
        SecretMetadata secret = getSecretMetadata(secretId);
        
        switch (secret.getState()) {
            case STABLE:
                return providedSecret.equals(secret.getCurrentKey());
                
            case ROTATION_START:
            case OVERLAP_WINDOW:
                // Accept BOTH old and new keys during overlap
                return providedSecret.equals(secret.getCurrentKey()) || 
                       providedSecret.equals(secret.getNewKey());
                       
            case ROTATION_COMPLETE:
                return providedSecret.equals(secret.getCurrentKey());
        }
        
        return false;
    }
}

Pull + Push Hybrid Distribution

Self-Healing Consistency Pattern

Combine push notifications with periodic pull for drift correction

// Consumer Agent with hybrid sync
public class HybridSecretSync {
    
    // Push-based: Real-time Kafka updates
    public void processKafkaEvent(SecretRotationEvent event) {
        logger.info("Received rotation event for: {}", event.getSecretId());
        
        // Immediately fetch and apply new secret
        applySecretRotation(event.getSecretId(), event.getNewVersion());
        
        // Update local tracking
        lastUpdateTime.put(event.getSecretId(), Instant.now());
    }
    
    // Pull-based: Periodic consistency check
    @Scheduled(fixedDelay = 300000) // every 5 minutes
    public void performConsistencyCheck() {
        for (String secretId : monitoredSecrets) {
            try {
                // Compare local version with Vault
                String localVersion = getLocalSecretVersion(secretId);
                String vaultVersion = vaultClient.getLatestVersion(secretId);
                
                if (!localVersion.equals(vaultVersion)) {
                    logger.warn("Drift detected - {} local={} vault={}", 
                               secretId, localVersion, vaultVersion);
                    
                    // Self-heal by pulling latest
                    pullAndApplySecret(secretId, vaultVersion);
                    
                    // Report drift metric for monitoring
                    metricsClient.increment("secret.drift.detected", 
                                          Map.of("secretId", secretId));
                }
            } catch (Exception e) {
                logger.error("Consistency check failed for {}", secretId, e);
            }
        }
    }
    
    // Emergency pull: Force sync all secrets
    @EventListener
    public void handleEmergencySync(EmergencySyncEvent event) {
        logger.warn("Emergency sync triggered by: {}", event.getReason());
        
        for (String secretId : monitoredSecrets) {
            try {
                String latestVersion = vaultClient.getLatestVersion(secretId);
                pullAndApplySecret(secretId, latestVersion);
                Thread.sleep(100); // Rate limit to avoid overwhelming Vault
            } catch (Exception e) {
                logger.error("Emergency sync failed for {}", secretId, e);
            }
        }
    }
}

Database Password Rotation

Complex Credential Handoff

Database passwords require careful connection pool management

// Database password rotation with connection pool handling
public class DatabasePasswordRotation {
    
    public void rotateDBPassword(String dbSecretId) {
        DatabaseConfig dbConfig = getDBConfig(dbSecretId);
        
        // Step 1: Create new database user with same permissions
        String newUsername = generateNewUsername(dbConfig.getServiceName());
        String newPassword = generateSecurePassword();
        
        createDatabaseUser(dbConfig.getHost(), newUsername, newPassword, 
                          dbConfig.getPermissions());
        
        // Step 2: Update secret in Vault with both users
        SecretData dualUserSecret = SecretData.builder()
            .currentUser(dbConfig.getCurrentUser())
            .currentPassword(dbConfig.getCurrentPassword())
            .newUser(newUsername)
            .newPassword(newPassword)
            .rotationState(OVERLAP_WINDOW)
            .build();
            
        vaultClient.updateSecret(dbSecretId, dualUserSecret);
        
        // Step 3: Notify all consumer services
        publishRotationEvent(dbSecretId, "DB_PASSWORD_ROTATED");
        
        // Step 4: Wait for services to update connection pools
        Thread.sleep(Duration.ofMinutes(5));
        
        // Step 5: Validate no connection errors
        if (validateDBConnections(dbSecretId)) {
            // Step 6: Drop old database user
            dropDatabaseUser(dbConfig.getHost(), dbConfig.getCurrentUser());
            
            // Step 7: Update secret to only contain new user
            SecretData finalSecret = SecretData.builder()
                .currentUser(newUsername)
                .currentPassword(newPassword)
                .rotationState(STABLE)
                .build();
                
            vaultClient.updateSecret(dbSecretId, finalSecret);
        } else {
            // Rollback: Drop new user and revert
            dropDatabaseUser(dbConfig.getHost(), newUsername);
            rollbackDatabaseRotation(dbSecretId);
        }
    }
    
    // Connection pool update in consuming service
    public void updateConnectionPool(String dbSecretId) {
        SecretData secret = vaultClient.getSecret(dbSecretId);
        
        if (secret.getRotationState() == OVERLAP_WINDOW) {
            // Gradual migration: new connections use new user
            connectionPool.setNewCredentials(
                secret.getNewUser(), 
                secret.getNewPassword()
            );
            
            // Existing connections continue with old user until natural expiry
            connectionPool.setGracefulMigration(true);
        } else {
            // Normal case: immediate update
            connectionPool.updateCredentials(
                secret.getCurrentUser(),
                secret.getCurrentPassword()
            );
        }
    }
}

TLS Certificate Rotation

Certificate Chain Validation

// TLS certificate rotation with chain validation
public class TLSCertificateRotation {
    
    public void rotateTLSCertificate(String certSecretId) {
        CertificateConfig certConfig = getCertificateConfig(certSecretId);
        
        // Step 1: Generate new certificate with same SAN
        CSR newCSR = generateCSR(certConfig.getCommonName(), 
                               certConfig.getSubjectAltNames());
        
        X509Certificate newCert = caClient.signCertificate(newCSR);
        PrivateKey newPrivateKey = newCSR.getPrivateKey();
        
        // Step 2: Validate certificate chain
        validateCertificateChain(newCert, caClient.getCACertificate());
        
        // Step 3: Update secret with both certificates
        TLSSecretData dualCertSecret = TLSSecretData.builder()
            .currentCert(certConfig.getCurrentCertificate())
            .currentPrivateKey(certConfig.getCurrentPrivateKey())
            .newCert(newCert)
            .newPrivateKey(newPrivateKey)
            .rotationState(OVERLAP_WINDOW)
            .overlapEnd(Instant.now().plus(Duration.ofDays(7))) // 7-day overlap
            .build();
            
        vaultClient.updateSecret(certSecretId, dualCertSecret);
        
        // Step 4: Notify services to update TLS configuration
        publishRotationEvent(certSecretId, "TLS_CERT_ROTATED");
        
        // Step 5: Validate TLS handshakes succeed
        if (validateTLSHandshakes(certSecretId)) {
            // Complete rotation
            completeCertificateRotation(certSecretId, newCert, newPrivateKey);
        } else {
            // Rollback certificate
            rollbackCertificateRotation(certSecretId);
        }
    }
    
    // Service TLS configuration hot-reload
    public void reloadTLSConfiguration(String certSecretId) {
        TLSSecretData secret = vaultClient.getSecret(certSecretId);
        
        if (secret.getRotationState() == OVERLAP_WINDOW) {
            // Configure server to accept both certificates
            tlsServer.addCertificate(secret.getNewCert(), secret.getNewPrivateKey());
            tlsServer.keepCertificate(secret.getCurrentCert()); // Keep old for clients
            
            // Client connections should prefer new certificate
            tlsClient.setPrimaryCertificate(secret.getNewCert(), secret.getNewPrivateKey());
            tlsClient.setFallbackCertificate(secret.getCurrentCert(), secret.getCurrentPrivateKey());
        } else {
            // Single certificate mode
            tlsServer.setCertificate(secret.getCurrentCert(), secret.getCurrentPrivateKey());
            tlsClient.setCertificate(secret.getCurrentCert(), secret.getCurrentPrivateKey());
        }
    }
}

Emergency Rollback Implementation

One-Click Rollback Mechanism

// Emergency rollback controller
public class EmergencyRollbackController {
    
    // One-click rollback to previous valid version
    public void emergencyRollback(String secretId, String reason, String operator) {
        SecretMetadata secret = getSecretMetadata(secretId);
        
        if (secret.getPreviousVersion() == null) {
            throw new IllegalStateException("No previous version available for rollback");
        }
        
        EmergencyRollbackEvent rollbackEvent = EmergencyRollbackEvent.builder()
            .secretId(secretId)
            .rollbackToVersion(secret.getPreviousVersion())
            .currentVersion(secret.getCurrentVersion())
            .reason(reason)
            .operator(operator)
            .timestamp(Instant.now())
            .build();
        
        // Immediately broadcast rollback to all consumers
        publishEmergencyEvent(secretId, rollbackEvent);
        
        // Update Vault state
        secret.setCurrentVersion(secret.getPreviousVersion());
        secret.setRotationState(STABLE);
        vaultClient.updateSecretMetadata(secretId, secret);
        
        // Alert security team
        alertManager.page("Emergency rollback executed for " + secretId + 
                         " by " + operator + " reason: " + reason);
        
        // Audit log
        auditLogger.critical("EMERGENCY_ROLLBACK executed for {} by {} reason: {}", 
                            secretId, operator, reason);
    }
    
    // Consumer agent processes emergency rollback
    public void processEmergencyRollback(EmergencyRollbackEvent event) {
        logger.warn("Processing emergency rollback for {}: {}", 
                   event.getSecretId(), event.getReason());
        
        // Immediately revert to previous version
        String previousSecret = vaultClient.getSecretVersion(
            event.getSecretId(), 
            event.getRollbackToVersion()
        );
        
        // Hot-reload service with previous secret
        writeSecretToFile(event.getSecretId(), previousSecret);
        signalServiceReload(event.getSecretId());
        
        // Confirm rollback success
        reportRollbackSuccess(event.getSecretId(), event.getRollbackToVersion());
    }
}

Page 4 — Security, Failure Scenarios & Cross-Colo Design

Security Failure Scenarios & Mitigation

Scenario 1: Partial Rotation (Authentication Split-Brain)

Failure: 30% of services update to new password, 70% still use old password

Impact: Intermittent authentication failures across user requests

Detection:

  • Auth success rate drops below 99.5% threshold
  • Consumer agents report version drift in health checks
  • Monitoring alerts on secret version inconsistency >5%

Mitigation:

  • **Dual-key overlap window** — Both old/new secrets valid for 15 minutes
  • **Automatic metric gates** — Rollback if auth success drops
  • **Emergency sync** — Force all agents to pull latest version
  • **Blast radius control** — Max 10 concurrent rotations per service tier

Recovery Time: 30 seconds (emergency rollback), 5 minutes (emergency sync)

Scenario 2: HSM Outage During Rotation

Failure: Hardware Security Module becomes unavailable mid-rotation

Impact: Cannot decrypt existing secrets or generate new ones

Mitigation:

  • **HSM High Availability** — Primary + standby HSM clusters across DCs
  • **Cached DEK (Data Encryption Keys)** — 1-hour cache for emergency decryption
  • **Rotation pause** — Automatically halt new rotations during HSM outage
  • **Cross-DC failover** — LVA1 HSM takes over from LTX1 within 60 seconds

Recovery Time: 60 seconds (HSM failover), 0 seconds (cached keys)

Scenario 3: Malicious Rotation Policy

Failure: Bad actor uploads policy "rotate every 10 seconds"

Impact: Self-inflicted DDoS on Vault and consumer services

Detection:**

  • Rate limit exceeded: >50 rotations/hour for single secret
  • Vault CPU/memory alerts due to encryption load
  • Consumer agents unable to keep up with rotation events

Prevention & Mitigation:

  • **Policy validation** — Min rotation period 1 hour, max concurrent rotations
  • **Rate limiting** — Per-service rotation quotas enforced by controller
  • **Policy approval** — Rotation policies require security team approval
  • **Circuit breaker** — Halt rotations if system load exceeds threshold

Recovery Time: 0 seconds (prevention), 30 seconds (circuit breaker)

Scenario 4: Cross-Region Network Partition

Failure: LTX1 completes rotation, LVA1 services miss update

Impact: Cross-region API calls fail due to credential mismatch

Detection:

  • Cross-region API success rate drops to 0%
  • Consumer agents in LVA1 report drift vs Vault primary
  • Kafka consumer lag spikes in affected regions

Mitigation:

  • **Regional Vault replicas** — Each DC has local Vault for reads
  • **Pull-based fallback** — Agents poll local Vault if Kafka fails
  • **Extended overlap window** — 30 minutes during network issues
  • **Manual resync** — Operations team can trigger regional sync

Recovery Time: 0 seconds (local reads), 5 minutes (manual sync)

Scenario 5: Consumer Agent Bug (Memory Leak)

Failure: Agent memory leak causes OOM, stops processing rotations

Impact: Affected services stuck with old secrets, eventual auth failure

Detection:

  • Agent health check fails (memory >90% of limit)
  • Secret version drift detected by monitoring
  • Service reports old secret version in health endpoint

Mitigation:

  • **Memory limits** — Agents run in containers with hard memory limits
  • **Automatic restart** — K8s restarts agent if health check fails
  • **Graceful degradation** — Service falls back to direct Vault calls
  • **Agent redundancy** — Multiple agents per service for HA

Recovery Time: 30 seconds (container restart), 0 seconds (direct Vault)

Cross-DC Failover Implementation

// Cross-datacenter failover controller
public class CrossDCSecurityFailover {
    
    private enum DCSecurityStatus { HEALTHY, DEGRADED, COMPROMISED }
    private final Map<String, DCSecurityStatus> dcSecurityMap = new ConcurrentHashMap<();
    
    // Continuous security monitoring
    @Scheduled(fixedDelay = 15000) // every 15s (faster than feature flags)
    public void monitorDatacenterSecurity() {
        for (String dc : Arrays.asList("ltx1", "lva1", "ei4", "grid2")) {
            DCSecurityStatus status = checkDatacenterSecurityHealth(dc);
            DCSecurityStatus oldStatus = dcSecurityMap.put(dc, status);
            
            if (status == DCSecurityStatus.COMPROMISED && oldStatus != DCSecurityStatus.COMPROMISED) {
                handleSecurityIncident(dc);
            }
        }
    }
    
    private void handleSecurityIncident(String compromisedDC) {
        switch (compromisedDC) {
            case "ltx1":
                // Primary DC compromised - immediate failover
                emergencyFailoverToPrimary("lva1");
                revokeAllSecretsInDC("ltx1");
                alertSecurityTeam("PRIMARY_DC_COMPROMISED", compromisedDC);
                break;
                
            case "lva1":
                // Hot standby compromised - isolate and rotate
                isolateDatacenter("lva1");
                triggerEmergencyRotationWave();
                alertSecurityTeam("STANDBY_DC_COMPROMISED", compromisedDC);
                break;
                
            default:
                // Other DC compromised
                isolateDatacenter(compromisedDC);
                auditSecretsInDC(compromisedDC);
                alertSecurityTeam("DC_COMPROMISED", compromisedDC);
        }
    }
    
    private DCSecurityStatus checkDatacenterSecurityHealth(String dc) {
        try {
            // Check HSM tamper detection
            boolean hsmHealthy = hsmClient.checkTamperStatus(dc);
            // Check Vault seal status
            boolean vaultSealed = vaultClient.isSealStatus(dc);
            // Check for unusual access patterns
            boolean anomalousAccess = detectAnomalousAccess(dc);
            // Check network intrusion detection
            boolean networkSecure = networkSecurityCheck(dc);
            
            if (!hsmHealthy || anomalousAccess) {
                return DCSecurityStatus.COMPROMISED;
            } else if (vaultSealed || !networkSecure) {
                return DCSecurityStatus.DEGRADED;
            } else {
                return DCSecurityStatus.HEALTHY;
            }
        } catch (Exception e) {
            // Security check failure = assume compromised
            return DCSecurityStatus.COMPROMISED;
        }
    }
    
    // Emergency secret revocation across entire DC
    private void revokeAllSecretsInDC(String datacenter) {
        logger.critical("Revoking ALL secrets in datacenter: {}", datacenter);
        
        List<String> allSecrets = vaultClient.listAllSecrets();
        
        allSecrets.parallelStream().forEach(secretId -> {
            try {
                // Revoke current version
                vaultClient.revokeSecretVersion(secretId, "EMERGENCY_DC_COMPROMISE");
                
                // Generate new version in healthy DC
                if (!datacenter.equals("ltx1")) {
                    rotationController.emergencyRotate(secretId, "DC_COMPROMISE_" + datacenter);
                }
            } catch (Exception e) {
                logger.error("Failed to revoke secret {} during DC compromise", secretId, e);
            }
        });
    }
}

// Secret classification and governance policy
public enum SecretClassification {
    PUBLIC,           // No rotation required
    INTERNAL,         // 180-day rotation
    CONFIDENTIAL,     // 90-day rotation  
    RESTRICTED,       // 30-day rotation
    TOP_SECRET        // 24-hour rotation
}

public class SecretGovernancePolicy {
    
    public RotationPolicy getRotationPolicy(String secretId) {
        SecretClassification classification = classifySecret(secretId);
        
        return switch (classification) {
            case PUBLIC -> RotationPolicy.noRotation();
            
            case INTERNAL -> RotationPolicy.builder()
                .rotationPeriod(Duration.ofDays(180))
                .overlapWindow(Duration.ofHours(24))
                .approvalRequired(false)
                .build();
                
            case CONFIDENTIAL -> RotationPolicy.builder()
                .rotationPeriod(Duration.ofDays(90))
                .overlapWindow(Duration.ofHours(2))
                .approvalRequired(false)
                .metricGates(List.of("auth_success_rate >= 99.5%"))
                .build();
                
            case RESTRICTED -> RotationPolicy.builder()
                .rotationPeriod(Duration.ofDays(30))
                .overlapWindow(Duration.ofMinutes(15))
                .approvalRequired(true)
                .emergencyContacts(List.of("security-oncall", "compliance-team"))
                .auditLevel(ENHANCED)
                .build();
                
            case TOP_SECRET -> RotationPolicy.builder()
                .rotationPeriod(Duration.ofHours(24))
                .overlapWindow(Duration.ofMinutes(5))
                .approvalRequired(true)
                .emergencyContacts(List.of("ciso", "security-oncall"))
                .auditLevel(MAXIMUM)
                .hsmRequired(true)
                .build();
        };
    }
    
    private SecretClassification classifySecret(String secretId) {
        // Classification based on secret metadata and service context
        if (secretId.contains("payment") || secretId.contains("financial")) {
            return TOP_SECRET;
        } else if (secretId.contains("user-data") || secretId.contains("pii")) {
            return RESTRICTED;
        } else if (secretId.contains("internal-api")) {
            return CONFIDENTIAL;
        } else if (secretId.contains("test") || secretId.contains("staging")) {
            return INTERNAL;
        } else {
            return CONFIDENTIAL; // Default to higher security
        }
    }
}

            "serviceId", serviceId,

        }

Runbook: High Authentication Failure Rate

Alert: "Auth success rate <99.9% for 2 minutes"

Investigation Steps:

  1. Check if rotation is in progress: kubectl get rotations --field-selector status=in-progress
  2. Identify affected secret: Review alert tags for secretId
  3. Check consumer agent health: Verify agents are receiving updates
  4. Validate HSM status: Ensure encryption/decryption is working
  5. Review recent deployments: Look for consumer service changes

Mitigation:

  • If rotation in progress: Extend overlap window by 15 minutes
  • If consumer drift: Execute emergency sync command
  • If HSM issues: Failover to standby HSM cluster
  • If critical: Execute emergency rollback to previous version

Runbook: Secret Drift Above Threshold

Alert: "Secret drift >0.5% detected"

Investigation Steps:

  1. Identify drifted services: curl /metrics/secret-drift | grep affected_services
  2. Check Kafka consumer lag: Look for stuck consumers
  3. Verify network connectivity: Test cross-DC communication
  4. Review consumer agent logs: Look for pull/push failures
  5. Check Vault replication status: Ensure cross-DC sync working

Mitigation:

  • Restart stuck consumer agents on affected services
  • Execute manual pull sync: kubectl exec agent -- sync-secrets --force
  • Reset Kafka consumer offsets if messages were lost
  • Trigger emergency broadcast if >10% drift

Runbook: HSM Tamper Alert

Alert: "HSM tamper detection triggered in ltx1"

Immediate Actions (First 60 seconds):

  1. **STOP ALL ROTATIONS** — Halt rotation controller immediately
  2. **ISOLATE AFFECTED HSM** — Block network access to compromised HSM
  3. **ENGAGE SECURITY TEAM** — Page CISO and security oncall
  4. **PRESERVE EVIDENCE** — Lock HSM state for forensic analysis

Recovery Actions:

  • Failover to standby HSM in LVA1 for urgent operations
  • Audit all secrets accessed in last 24 hours
  • Plan coordinated rotation wave for all potentially exposed secrets
  • Conduct forensic analysis before re-enabling compromised HSM

// Immutable audit trail with cryptographic integrity
public class SecretAuditService {
    
    // Record every secret access with hash chaining
    public void auditSecretAccess(SecretAccessEvent event) {
        AuditRecord record = AuditRecord.builder()
            .recordId(UUID.randomUUID().toString())
            .timestamp(Instant.now())
            .eventType(event.getEventType()) // READ, WRITE, ROTATE, DELETE
            .secretId(event.getSecretId())
            .secretVersion(event.getSecretVersion())
            .accessor(event.getServiceId())
            .userContext(event.getUserId())
            .sourceIP(event.getSourceIP())
            .datacenter(event.getDatacenter())
            .success(event.isSuccess())
            .failureReason(event.getFailureReason())
            .previousRecordHash(getLastRecordHash())
            .build();
        
        // Calculate cryptographic hash including previous record
        String recordHash = calculateAuditHash(record);
        record.setRecordHash(recordHash);
        
        // Store in multiple locations for integrity
        kafkaProducer.send("audit-log", record.getSecretId(), record);
        espressoClient.storeAuditRecord(record);
        
        // Real-time compliance monitoring
        complianceMonitor.processAuditRecord(record);
        
        // Alert on suspicious patterns
        if (detectSuspiciousAccess(record)) {
            alertManager.page("SUSPICIOUS SECRET ACCESS: {}", record);
        }
    }
    
    // Cryptographic integrity verification
    private String calculateAuditHash(AuditRecord record) {
        String hashInput = record.getSecretId() + ":" +
                          record.getEventType() + ":" + 
                          record.getTimestamp() + ":" +
                          record.getAccessor() + ":" +
                          record.getPreviousRecordHash();
        
        return DigestUtils.sha256Hex(hashInput);
    }
    
    // Verify audit chain integrity (compliance requirement)
    public ComplianceReport verifyAuditIntegrity(String secretId, 
                                                Duration timeRange) {
        List<AuditRecord> records = getAuditHistory(secretId, timeRange);
        
        ComplianceReport report = new ComplianceReport();
        report.setSecretId(secretId);
        report.setTimeRange(timeRange);
        report.setTotalRecords(records.size());
        
        // Verify hash chain
        for (int i = 1; i < records.size(); i++) {
            AuditRecord current = records.get(i);
            AuditRecord previous = records.get(i - 1);
            
            if (!current.getPreviousRecordHash().equals(previous.getRecordHash())) {
                report.addIntegrityViolation("Broken hash chain at record " + i);
            }
        }
        
        // Verify completeness (no gaps in timestamps)
        for (int i = 1; i < records.size(); i++) {
            Duration gap = Duration.between(records.get(i-1).getTimestamp(), 
                                          records.get(i).getTimestamp());
            if (gap.toMinutes() > 60) {
                report.addComplianceIssue("Audit gap detected: " + gap);
            }
        }
        
        return report;
    }
    
    // Suspicious access pattern detection
    private boolean detectSuspiciousAccess(AuditRecord record) {
        // Multiple rapid accesses from different IPs
        if (countRecentAccesses(record.getSecretId(), Duration.ofMinutes(5)) > 100) {
            return true;
        }
        
        // Access from unusual datacenter
        if (!record.getDatacenter().matches("ltx1|lva1|ei4")) {
            return true;
        }
        
        // Failed access attempts followed by success
        if (record.isSuccess() && 
            countRecentFailures(record.getSecretId(), Duration.ofMinutes(1)) > 10) {
            return true;
        }
        
        return false;
    }
}
